# Encryption Reference

This document covers the **only reversible encryption path** in the codebase: `CryptoService.encryptToken` / `decryptToken`, used by `VaultPasswordService` to protect the `vaultCredentialVerifier` field on User documents.

Explicitly out of scope (not documented here because they are not encrypted):

- **File contents** — uploaded and downloaded via AWS S3 presigned URLs. The backend never sees the bytes and nothing is encrypted at the application layer.
- **User passwords** — one-way hashed with `bcryptjs` (12 rounds). Not reversible, no decryption exists.
- **JWT access/refresh tokens** — signed (HS256 via `jose`), not encrypted. Refresh tokens are SHA-256 hashed before DB storage.

---

## Algorithm and Library

| Property | Value |
|---|---|
| Library | `crypto-js` (`CryptoJS.AES.encrypt` / `CryptoJS.AES.decrypt`) |
| Cipher | AES-256-CBC with PKCS#7 padding (CryptoJS default in passphrase mode) |
| KDF | OpenSSL `EVP_BytesToKey` (MD5-based) — CryptoJS legacy default when a string passphrase is provided |
| Salt | Random 8 bytes per call, generated by CryptoJS |
| IV | Derived alongside the key by the KDF |
| Output format | OpenSSL `Salted__…` envelope, base64-encoded string |
| Authenticated? | **No.** No HMAC, no AEAD tag. See limitations below. |

## Key Material

- **Source:** `CRYPTO_SECRET` environment variable.
- **Validation:** `z.string().min(32)` in `src/config/env.validation.ts` — the process refuses to start if shorter.
- **Scope:** Single global passphrase. Not per-user, not per-record, not versioned.
- **Rotation:** Not supported. Changing `CRYPTO_SECRET` invalidates every existing `vaultCredentialVerifier` value because they cannot be decrypted with the new secret.

## Encryption Flow

Called from `VaultPasswordService.setVerifier` the first time a user establishes a vault credential. The plaintext input is an already-encrypted verifier token supplied by the client (the server never sees the raw vault password itself).

```
client-supplied encrypted token
          │
          ▼
CryptoService.encryptToken(token)
          │
          ├─ CryptoJS.AES.encrypt(token, CRYPTO_SECRET)
          │    random 8-byte salt + IV derived by KDF
          │
          ▼
 base64 "Salted__…" ciphertext
          │
          ▼
 User.vaultCredentialVerifier  (MongoDB, users collection)
```

If the field is already set, `setVerifier` does **not** overwrite. It decrypts the existing value and returns `{ alreadySet: true, credentialChecker }` so the caller can compare against the submitted value.

## Decryption Flow

Called from `VaultPasswordService.getVerifier` whenever the verifier is needed.

1. Load `user.vaultCredentialVerifier` from MongoDB.
2. Call `CryptoService.decryptToken(ciphertext)`.
   - Internally runs `CryptoJS.AES.decrypt(ciphertext, CRYPTO_SECRET)` and converts the result to UTF-8.
   - Any exception or empty output → returns `null` (errors are swallowed inside the helper).
3. Caller interprets `null` as corruption and throws `InternalServerErrorException('Vault password corrupted.')`.
4. On success, the UTF-8 string is returned to the caller.

Missing user → `NotFoundException('User not found.')`.
Missing/empty verifier → `NotFoundException('No vault password set.')`.

## Consumers

Only two methods in the repo call the crypto primitives. No other code path uses `encryptToken` / `decryptToken`.

| Method | Purpose |
|---|---|
| `CryptoService.encryptToken(plaintext: string): string` | Encrypt helper. |
| `CryptoService.decryptToken(ciphertext: string): string \| null` | Decrypt helper. Swallows errors, returns `null` on failure. |
| `VaultPasswordService.setVerifier(userId, encryptedToken)` | First-write path. Idempotent: if a verifier already exists, returns the existing decrypted value instead of overwriting. |
| `VaultPasswordService.getVerifier(userId)` | Read path. Returns the decrypted verifier or raises. |

## Stored Field

Defined in `src/shared/database/schemas/user.schema.ts`:

```ts
@Prop({ default: '' })
vaultCredentialVerifier: string;
```

Stored as a plain string on the `users` collection. Default is the empty string; `getVerifier` treats empty as "not set".

## Dependencies

- Runtime: `crypto-js ^4.2.0`, `@nestjs/common`, `@nestjs/config`, `@nestjs/mongoose`, `mongoose`.
- Env: `CRYPTO_SECRET` (≥ 32 chars).
- Not used: `node:crypto` for this path, `libsodium`, KMS, HSM.

## Known Limitations

- **Unauthenticated cipher.** CBC without an HMAC/AEAD tag means ciphertext tampering is not detected on decrypt. A tampered value either yields garbage or triggers the generic "corrupted" error — there is no explicit integrity check.
- **Legacy KDF.** `EVP_BytesToKey` with MD5 is weak compared to PBKDF2 or Argon2. Acceptable here because the input is a ≥ 32-char high-entropy secret from configuration, but worth flagging for any future hardening.
- **Single global key, no rotation.** Leaking `CRYPTO_SECRET` exposes every verifier in the database, and rotation requires re-encrypting every row in a migration.
- **Not suitable for files or binary data.** CryptoJS is synchronous and string-oriented; large or binary payloads should use `node:crypto` streaming AES-GCM instead. This path is scoped to short verifier strings.
